Build the SDD: /v1 → /s1 + the Paragraph That's Yours

Ciao — this is Kore. Episode 2 of 4. Run the sequence and let Gru push. The Software Design Document — the SDD — is only worth grading if it shows the pushback you resolved. Don't clean it up. The flags Gru raises are the evidence of your thinking. A polished SDD with no flags is a document that was easy to write. That's not what this assignment is measuring. /v1 is the problem summary. It must distinguish this system from a generic one — which means it names comparable systems and states the success condition for this specific deployment. 'A system that flags insurance claims' does not pass /v1 if there are a dozen comparable systems. Your /v1 must say what makes this system's success condition different from those. After /v1 passes, write the personal paragraph before running /v2. Three to five sentences in your own voice: what Gru's pushback changed about how you understood the problem. This is the only thing you add to the SDD — everything else is Gru's output, submitted unedited. /v2 is the design principles: three to four non-negotiable principles with the collision test completed for each. The collision test is not optional — it is the requirement that distinguishes a principle from a preference. A preference is 'the system should be accurate.' A principle with a collision test is 'accuracy is non-negotiable even when it increases latency above the stated SLA, because a missed violation is more costly than a delayed queue entry.' That sentence states what you sacrifice when the principle is honored.

What you'll see:
• /v3 is the core flows. Every step must be concrete. Never write 'the system processes the request' — that sentence names nothing and specifies nothing. Name the actor, the input, the transform, and the output at every step. A concrete flow step reads: 'The claim intake service receives a structured JSON payload from the intake form. The policy-check module applies rule set v3.2 to the claim fields. If the rule set finds a violation, the module writes a violation record to the audit queue with the field name, the rule ID, and the rejection reason.' A vague flow step reads: 'The system checks the claim and decides whether to flag it.' The second version cannot be tested. /v4 is the need statements: five to eight testable needs, each with a pass/fail condition that is specific enough to run as a test case. Not 'the system should handle edge cases' — that is not testable. Instead: 'The system rejects a claim where the vendor ID field is present but maps to no record in the approved-vendor list, and logs the rejection reason as UNKNOWN_VENDOR.' That sentence could be a test in a test suite. Write needs at that level of specificity.
• /s1 assembles the Software Design Document. At this gate, Gru maps every component to a Need from /v4 and checks every component's specification: inputs, outputs, edge cases, and scope boundary. The key trap: a component with no Need mapped to it is scope creep. Gru will flag it. When it does, your options are: remove the component, or write a new Need in /v4 that justifies it and re-run the gate. The second trap — and this one loses more points — is cosmetic cleanup of Gru's flags. The flags are not formatting problems. They are substantive gaps in your specification. When Gru says a component's edge cases are underspecified, the correct response is to specify the edge cases, not to rewrite the flag in smoother language. The grader is reading for evidence that you resolved the flags by improving the specification. A clean SDD that was cleaned by editing the flag text is not an SDD — it is a coverup. Submit Gru's unedited output plus your personal paragraph. That is the complete submission.
• The personal paragraph is three to five sentences, written after /v1, before /v2. It belongs to you — Gru does not write it and you do not ask Gru to write it. The question it answers: what did Gru's pushback change about how you understood the problem? Not what Gru said — what changed in your understanding as a result. If Gru accepted your first /v1 formulation without pushback, your paragraph must address that too: was the formulation genuinely tight, or did you get lucky? 'I got lucky' is a valid answer if you can say why. This is the only place in the SDD where your voice appears. The grader uses it to distinguish students who ran the gates from students who understood what the gates were doing. Three to five sentences. Specific. First person. No hedging.
• Verdict for Build the SDD. Run in order: /v1 through /s1. Write the personal paragraph after /v1, before /v2 — three to five sentences, first person, yours alone. /v1 distinguishes this system from generic comparables. /v2 has three to four principles, each with a collision test. /v3 flows name actor, input, transform, and output at every step. /v4 has five to eight testable needs, each a valid test case. /s1 maps every component to a Need. Two traps: component with no Need is scope creep; cosmetic cleanup of flags is a coverup. Submit Gru's output unedited plus the paragraph. That is the SDD.
